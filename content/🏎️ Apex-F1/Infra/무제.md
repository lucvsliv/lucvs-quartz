### 멈춰버린 EC2
![[스크린샷 2026-03-21 18.40.10.png]]
Spring Boot 프로젝트를 가져와서 Docker Compose를 하려고 하니까 갑자기 빌드 도중에 EC2가 아예 멈춰버렸다. 지난번에는 Docker를 사용하지 않고 그냥 바로 Spring Application을 실행하니까 CPU 사용량이 최대치를 초과하여 멈춘 적은 있지만 빌드 과정에서는 처음이다.


![[스크린샷 2026-03-21 18.39.32.png]]

EC2 인스턴스 모니터링을 보니 아직 모두 최대치는 도달하지 않았지만 비슷한 이유로 인스턴스가 멈춘 것 같다. 가장 큰 이유는 OOM(Out of Memory)인데, AWS EC2 `t2.micro`의 기본 메모리는 `1GB`밖에 되지 않기 때문이다.

### Swap 메모리를 만들자
운영체제 수업에서 XV6 과제 중에 Memory Swap을 직접 구현하던 기억이 난다. 빈 메모리 영역을 비트값으로 관리하고, 이 비트맵에서 연속적/비연속적으로 비어있는 영역을 찾아 해당 영역에 메모리를 할당했던 것 같다. 거기에 Atomicity를 위한 Lock까지... 

XV6 이야기는 여기까지 하고 스왑 메모리를 만들어보자.

![[스크린샷 2026-03-21 18.59.17.png]]
우선 `dd` command를 사용해서 디스크 용량을 파일처럼 다루면 된다. Input File로 `/dev/zero`를 사용하는데, `/dev/null`은 메모리 영역을 읽어오는 대신, 모두 `0`으로 초기화하여 읽어온다. 이 연속적인 `0` 스트림에서, `128MB`짜리 Block을 16개, 즉 `2048MB` 크기의 파일을 `/swapfile` 이라는 Output File로 만든다.

이 Swap File이 만들어지면, 적절한 권한을 부여하고, `/swapfile` 파일을 Swap 메모리로 포맷한다. 마지막으로 이 Swap 메모리를 활성화시켜주면 된다.

![[스크린샷 2026-03-21 18.59.31.png]]
재부팅 시에도 이 Swap 메모리가 유지될 수 있도록 해주고, 마지막으로 메모리 영역을 확인해보면 `2GB` 크기의 Swap 메모리 영역이 성공적으로 생긴 것을 알 수 있다.
