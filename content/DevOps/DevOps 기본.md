## Infrastructure/Deployment 개념

**1. CI / CD (지속적 통합 / 지속적 배포)**
- **CI (Continuous Integration):** 코드를 깃허브에 올렸을 때, 봇(GitHub Actions)이 알아서 코드를 다운받고, 아까 우리가 짠 **테스트 코드들을 전부 실행해 보고 이상이 없는지 검사**하는 과정입니다. "합격!" 판정이 나면 코드를 실행 가능한 파일(.jar)로 조립(Build)합니다. (자동화된 깐깐한 QA 요원 역할)
- **CD (Continuous Deployment):** 조립이 완료된 파일을 실제 서비스가 돌아가는 **서버(EC2 등)로 안전하게 배달하고 실행**시키는 과정입니다. (자동화된 택배 기사 역할)

**2. Docker (도커)**
- "내 컴퓨터에선 잘 되는데 서버에선 왜 안 되지?" 라는 개발자의 영원한 변명을 없애준 구세주입니다.
- 우리의 Spring Boot 앱, 자바(JDK), 설정 파일들을 **하나의 네모난 컨테이너 박스(이미지)**에 포장합니다. 이 박스는 Mac이든, Windows든, AWS 리눅스 서버든 도커만 깔려 있으면 **무조건 똑같이 실행**됩니다.

**3. 무중단 배포 (Blue/Green)와 Nginx**
- **기존 배포의 문제점:** 새 버전을 띄우려면 기존 서버를 끄고 켜야 합니다. 이때 사용자는 약 10~30초간 "서버 접속 불가(에러)" 화면을 보게 됩니다. (다운타임)
- **Blue/Green 해결책:** 서버 안에 똑같은 공간을 2개 만듭니다. (파란 방, 초록 방). 현재 유저들은 '파란 방(Blue)'에서 서비스를 이용 중입니다. 우리는 몰래 '초록 방(Green)'에 새 버전의 서버를 띄워둡니다.
- **Nginx(리버스 프록시):** 얘는 문지기입니다. 초록 방 서버가 완전히 켜져서 준비가 끝났다는 신호를 받으면, Nginx가 유저들이 들어오는 문을 파란 방에서 **초록 방으로 0.1초 만에 딸깍!** 하고 바꿔버립니다. 유저는 서버가 업데이트되었는지도 모르게 자연스럽게 새 버전을 쓰게 됩니다. 이게 **무중단 배포**입니다.
